# 球队别名全局匹配设计

## 背景

比赛 `14073`（汉坎对特罗姆瑟）和 `14972`（武汉三镇对深圳新鹏城）在拉取 The Odds API 赔率时未能匹配事件。历史事件返回的球队名称分别为 `HamKam`、`Tromso`、`Wuhan Three Towns` 和 `Shenzhen Peng City FC`。

仓库已经配置了以下别名：

- `Ham-Kam -> HamKam`
- `Sichuan Jiuniu -> Shenzhen Peng City`

这些别名标记为来自 `oddspapi`，而 The Odds API 拉取器只加载 `source_name: the_odds_api` 的别名，因此没有使用它们。球队的改名、缩写和拼写变体通常不是单一赔率数据源专属信息，按数据源隔离会造成重复配置和漏配。

## 范围

本次只将 `entity_type: team` 的别名用于所有赔率数据源的比赛匹配。联赛及其他实体的别名行为保持不变。

本次不重新拉取比赛 `14073` 和 `14972` 的赔率，也不修改展示名称配置。

## 方案

在赔率比赛匹配阶段构建全局球队别名池：

1. 从 `config/external_aliases.yaml` 加载全部球队别名，不按 `source_name` 过滤。
2. 从 `external_aliases` 数据表加载全部球队别名，不按 `source_name` 过滤。
3. 合并配置与数据库记录，按标准球队名和别名去重。
4. OddsPapi 和 The Odds API 拉取器使用同一全局球队别名池。
5. `source_name` 字段保留，用于记录别名来源和保持现有配置、数据库及 CLI 兼容；对球队赔率匹配而言，它不再限制适用的数据源。

不引入新的数据库字段或迁移，也不批量改写现有 YAML。这样可以直接复用当前 102 条配置球队别名和 4 条数据库球队别名。

## 冲突处理

加载全局球队别名池时，以现有名称标准化规则处理别名。如果同一个标准化别名映射到不同的标准球队，立即抛出包含冲突信息的明确错误，不静默选择其中一条。

相同标准球队与相同标准化别名的重复记录视为同一别名并去重。当前配置与数据库合计 106 条球队别名，审计未发现跨数据源映射冲突。

## 数据流

赔率拉取器启动单次同步时，从配置和数据库读取全局球队别名池，并把它传给现有比赛匹配函数。匹配函数仍以标准球队名为第一候选，同时计算其所有别名与外部事件球队名的最高相似度。比赛时间、联赛映射和置信度阈值均保持现有行为。

因此：

- `Ham-Kam` 使用已有别名 `HamKam`，与 The Odds API 名称完全匹配。
- `Sichuan Jiuniu` 使用已有别名 `Shenzhen Peng City`；现有名称标准化会移除末尾 `FC`，因此它与 `Shenzhen Peng City FC` 获得 `1.0` 相似度，无需增加带 `FC` 的重复别名。

## 代码边界

球队别名的配置读取、数据库读取、合并、去重和冲突检查应集中在共享服务中。`oddspapi_sync_runner` 和 `the_odds_api_sync_runner` 只负责调用共享加载逻辑，不各自维护过滤规则。

现有通用别名列表和 CLI 的按 `source_name` 查询能力继续保留，避免影响诊断和来源追踪。全局语义只作用于赔率拉取时的球队匹配输入。

## 测试与验收

自动化测试覆盖：

- 共享加载逻辑同时读取 YAML 和数据库中的球队别名。
- 标记为 `oddspapi` 来源的球队别名可被 The Odds API 匹配使用。
- 标记为 `the_odds_api` 来源的球队别名可被 OddsPapi 匹配使用。
- 重复的等价别名被去重。
- 同一标准化别名映射到不同球队时明确失败。
- `Ham-Kam` 对 `HamKam` 的回归匹配成功。
- `Sichuan Jiuniu` 对 `Shenzhen Peng City FC` 的回归匹配成功。

验收时运行球队别名服务、OddsPapi 同步器和 The Odds API 同步器的针对性测试。由于用户暂不要求重新拉取，本次验证不调用外部赔率 API，也不写入两场比赛的赔率快照。
