# 公司环境并行开发任务

## 背景

家里环境已经具备真实 API key、本地 SQLite 数据库和 OddsPapi 后台 worker，可以持续做历史比赛与赔率快照回填。公司环境可以从 GitHub 拉取代码继续开发，但默认没有 `.env`、API key、真实 SQLite 数据库和回填日志。

因此，公司环境的最佳价值不是继续跑数据，而是推进不依赖真实数据的工程能力：Web 控制台、回测框架、模型接口、评估指标、资金与信心等级策略、mock 数据展示和单元测试。

## 协作规则

- 设计文档、计划文档、提交记录使用中文。
- Python 代码、模块名、函数名和类名使用英文。
- 不提交 `.env`、API key、SQLite 数据库、日志文件、回填产物。
- 不在公司环境运行真实 API-Football、OddsPapi 或 The Odds API 请求。
- 不把测试改成依赖外部网络。
- 不修改家里正在运行的赔率回填批次语义，除非有明确文档和测试。
- 新功能优先用 mock 数据、内存数据库、测试夹具验证。

## 当前项目进度摘要

家里环境已经完成或正在推进：

- API-Football 比赛基础数据同步。
- 本地 SQLite 数据库落地。
- 主流联赛白名单和中文展示名。
- OddsPapi Pinnacle 单源历史赔率接入。
- 动态主盘口抽取：亚盘和大小球盘口按临场主盘口追踪。
- 开赛前 24 小时赔率快照压缩采样。
- OddsPapi 后台 worker、批量回填配置、完成通知。
- 未匹配比赛记录为终态，避免重复请求同一场比赛。
- Dixon-Coles、Skellam、负二项等模型骨架已经进入代码库。

当前主要缺口：

- 赔率覆盖率和未匹配情况缺少清晰仪表盘。
- CLI 命令已经较多，需要 Web 控制台降低操作负担。
- 回测框架还没有系统化。
- 模型输出、信心等级、手数策略还没有形成统一闭环。
- 赔率走势还没有正式沉淀为训练特征。

## 家里环境优先事项

这些任务依赖真实 API key、真实数据库或正在运行的 worker，应留在家里环境处理：

1. 继续 OddsPapi 历史赔率回填。
2. 观察 worker 状态和请求消耗。
3. 核对真实赔率覆盖率、失败原因和未匹配比赛。
4. 维护真实球队别名映射。
5. 基于真实数据库生成联赛覆盖率报告。
6. 用真实样本进行大规模训练和回测。

## 公司环境优先任务

### P0：Web 控制台一期 mock 骨架

目标：先做可运行的本地 Web 控制台，用 mock 数据呈现未来真实界面的结构。

范围：

- 使用 React/Vite 或项目确认的现代前端方案。
- 首屏直接是控制台，不做营销落地页。
- 页面包括：
  - 总览：比赛数、赔率覆盖率、worker 状态、待处理未匹配数。
  - 联赛覆盖率表：联赛、赛季、已完赛场次、有赔率场次、覆盖率。
  - worker 状态列表：pid、批次、联赛、状态、请求数、日志路径。
  - 单场比赛详情占位：基本信息、亚盘走势、大小球走势。
  - 推荐记录占位：信心等级、手数、盘口、结果。
- 数据全部来自 mock JSON 或前端 fixture。
- 预留后续真实 API 的数据结构，不接真实数据库。

验收：

- 本地能启动页面。
- 主要页面不依赖 `.env` 和 SQLite。
- 至少有一个赔率曲线 mock 图。
- 页面文字使用中文，模型字段可保留英文。
- 有基础构建或测试命令。

### P0：回测指标与报表核心

目标：先把回测结果的统计口径做成纯函数模块，后续真实数据接入时直接复用。

范围：

- 定义回测输入结构：
  - match_id
  - league_id
  - kickoff_time_beijing
  - market_type
  - line
  - side
  - odds
  - confidence_grade
  - stake_units
  - settlement_result
  - profit_units
- 实现统计指标：
  - 命中率
  - 走水率
  - ROI
  - yield
  - 总手数
  - 最大回撤
  - 按联赛分组
  - 按盘口类型分组
  - 按信心等级分组
- 使用纯单元测试和合成样本。

验收：

- 不依赖数据库。
- 不调用外部 API。
- 测试覆盖 win、half_win、push、half_loss、loss。
- 输出可被 Web 控制台直接展示。

### P1：盘口结算与训练标签模块

目标：把亚盘、大小球盘口输赢判定做成稳定、可复用、可测试的底层模块。

范围：

- 支持标准盘口：
  - `X`
  - `X.25`
  - `X.5`
  - `X.75`
- 亚盘支持主队让球、主队受让、客队方向。
- 大小球支持大球、小球方向。
- 输出：
  - win
  - half_win
  - push
  - half_loss
  - loss
  - profit_units
- 非标准盘口直接标记 invalid 或抛出可解释错误。

验收：

- 参数化测试覆盖常见盘口。
- 与现有推荐记录/回测输入结构兼容。
- 不依赖真实比赛数据。

### P1：信心等级与手数策略模块

目标：把 edge、模型概率、赔率价值映射为 D/C/B/A/S 等级和 0.5 到 3 手建议。

初版建议：

- `D`：观察，不出手或 0.5 手。
- `C`：低信心，0.5 到 1 手。
- `B`：常规信心，1 到 1.5 手。
- `A`：较高信心，1.5 到 2 手。
- `S`：高信心，2 到 3 手，需严格限制触发条件。
- 支持 `B+`、`A-` 这种微调等级。
- 最小单位为 0.25 手。

范围：

- 输入：
  - model_probability
  - market_implied_probability
  - edge
  - model_agreement_count
  - odds_quality_flag
  - recent_performance_adjustment
- 输出：
  - confidence_grade
  - stake_units
  - explanation
- 先做纯规则，不接真实资金账户。

验收：

- 有单元测试覆盖边界值。
- 最高等级有保护条件。
- 单场最大手数不超过 3。

### P1：统一模型输出接口整理

目标：让 Dixon-Coles、Skellam、负二项、未来机器学习模型都能进入同一推荐与回测管线。

范围：

- 梳理统一输出结构：
  - score_distribution
  - asian_handicap_probability
  - total_goals_probability
  - model_name
  - model_version
  - feature_snapshot
  - explanation_items
- 保持已有 `score_model_service.py` 思路。
- 不要求接真实训练数据。

验收：

- 有测试证明多个模型能输出统一结构。
- 推荐层可以消费统一结构。
- 文档说明各模型适用盘口。

### P2：批次配置执行器增强

目标：让 `config/oddspapi_backfill_batches.yaml` 不只是记录配置，而是可被 CLI 读取、dry-run 和启动。

范围：

- 新增批次配置读取服务。
- 校验字段：
  - name
  - season
  - from_date
  - league_ids
  - request_budget_per_league
  - max_rounds_per_league
- 新增命令建议：

```text
odds-source oddspapi-batches
odds-source oddspapi-batch-dry-run --name <批次名>
odds-source oddspapi-batch-start --name <批次名>
```

验收：

- 测试使用 fake starter，不启动真实 worker。
- YAML 错误有清晰中文提示。
- 不调用 OddsPapi。

### P2：worker 管理增强

目标：解决多个后台 worker 状态文件互相覆盖的问题，并为 Web 控制台提供数据源。

范围：

- 每个 worker 启动后写独立状态文件，例如 `logs/odds/workers/<pid>.json`。
- 新增所有 worker 列表命令。
- 新增单个 worker 日志尾部查看命令。
- 可选新增停止指定 worker 命令，测试里只 mock 进程，不杀真实进程。

验收：

- 多 worker 状态不会互相覆盖。
- 测试不依赖真实 worker。
- 输出字段适合 Web 控制台展示。

## 推荐分工

家里环境：

- 继续跑真实数据回填。
- 做真实覆盖率核对。
- 根据真实未匹配列表补别名。
- 后续用真实样本跑训练和回测。

公司环境：

- P0 优先做 Web 控制台 mock 骨架。
- P0 同步做回测指标纯函数模块。
- P1 做盘口结算、信心等级和统一模型输出接口。
- P2 再做批次配置执行器和 worker 管理增强。

## 推荐下一步

如果公司环境只有一条 Codex 会话，建议先做：

```text
P0：Web 控制台一期 mock 骨架
```

原因：

- 当前 CLI 已经很重，用户操作成本高。
- Web 控制台可以先用 mock 数据开发，不依赖真实 DB。
- 后续家里回填出的真实覆盖率、worker 状态、单场赔率曲线都能自然接入。

如果公司环境可以并行开两条线：

```text
会话一：Web 控制台一期 mock 骨架
会话二：回测指标与盘口结算模块
```

这两条线互不阻塞，且都不依赖真实 API 或本地数据库。
