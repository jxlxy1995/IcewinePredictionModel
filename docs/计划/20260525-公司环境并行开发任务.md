# 公司环境并行开发任务

## 背景

公司电脑可以从 GitHub 拉取工程继续开发，但默认没有本地 `.env`、API key 和家里机器上的 SQLite 数据库。公司环境应优先做不依赖真实 API 和真实数据库的大块功能，避免影响家里正在进行的赔率回填任务。

公司环境 Codex 开始前应先阅读：

- `docs/规则/20260523-130345-项目协作规则.md`
- 本文档
- 相关模块现有测试

## 环境假设

- 不存在 `.env`。
- 不存在 `local_data/icewine_prediction.sqlite3`。
- 不能访问家里机器正在回填的真实数据库。
- 不应使用真实 API-Football、OddsPapi 或 The Odds API key。
- 可以运行单元测试、使用内存数据库、使用测试夹具和 mock 数据。

## 禁止事项

- 不要提交 `.env`、API key、SQLite 数据库、日志文件或回填产物。
- 不要在公司环境运行真实 API 同步、真实赔率回填或真实大规模训练。
- 不要修改家里正在运行的回填批次语义，除非已经在文档中说明原因并有测试覆盖。
- 不要把测试改成依赖外部网络。

## 推荐任务一：Web 控制台骨架

目标：提前建设本地 Web 操作界面，降低后续 CLI 使用负担。

范围：

- 搭建 React/Vite 前端或后端内嵌静态页面骨架。
- 使用 mock 数据展示联赛覆盖率、回填批次、worker 状态、比赛列表、赔率曲线占位和推荐记录列表。
- 不接真实 API，不依赖真实 SQLite。
- 为后续真实接口预留清晰的数据结构。

验收：

- 本地能启动页面。
- 页面首屏是实际控制台，不是营销落地页。
- 有 mock 的回填状态、联赛覆盖率和赔率走势展示。
- 如有前端测试或构建命令，应在公司环境通过。

## 推荐任务二：后台 Worker 管理增强

目标：解决当前多个后台 worker 共用 `oddspapi-worker-current.json` 的状态覆盖问题。

范围：

- 每个 worker 启动后写独立状态文件，例如 `logs/odds/workers/<pid>.json`。
- 新增命令列出所有 worker：进程 pid、批次、联赛、状态、日志路径、最后更新时间。
- 新增查看单个 worker 日志尾部的命令。
- 可选：新增停止指定 worker 的命令，但测试中只 mock 进程，不杀真实进程。

建议命令名：

```text
odds-source oddspapi-workers
odds-source oddspapi-worker-show --pid <pid>
odds-source oddspapi-worker-stop --pid <pid>
```

验收：

- 使用单元测试 mock `subprocess.Popen` 和进程查询。
- 不依赖真实 OddsPapi。
- 多 worker 状态不会互相覆盖。

## 推荐任务三：批次配置执行器

目标：让 `config/oddspapi_backfill_batches.yaml` 不只是记录配置，而是可以被 CLI 直接执行。

范围：

- 新增读取批次配置的服务。
- 新增校验：批次名、season、league_ids、request_budget_per_league、max_rounds_per_league。
- 新增 dry-run 命令，输出会启动哪些批次和命令。
- 新增 start 命令，可按批次名启动一个或多个 worker。

建议命令名：

```text
odds-source oddspapi-batches
odds-source oddspapi-batch-dry-run --name 第一优先级A-北欧新赛季回填
odds-source oddspapi-batch-start --name 第一优先级A-北欧新赛季回填
```

验收：

- 不直接触发真实 API。
- 测试使用 fake starter 验证生成参数正确。
- YAML 格式错误时有清晰中文报错。

## 推荐任务四：盘口结算与训练样本逻辑

目标：完善亚盘和大小球历史样本的可训练标签。

范围：

- 亚盘支持 `X`、`X.25`、`X.5`、`X.75`。
- 大小球支持 `X`、`X.25`、`X.5`、`X.75`。
- 输出 win、half_win、push、half_loss、loss。
- 生成对应收益单位，便于回测。
- 非标准盘口直接舍弃或标记无效。

验收：

- 用参数化单元测试覆盖主队让球、客队受让、大小球大/小方向。
- 不依赖真实数据库。

## 推荐任务五：模型训练骨架增强

目标：在没有真实数据的公司环境，用合成样本推进模型层。

范围：

- 泊松进球模型。
- Dixon-Coles 修正模型骨架。
- 负二项进球分布骨架。
- 时间衰减权重。
- 训练/验证切分和评估指标。

验收：

- 使用小型合成样本测试。
- 输出模型参数、预测概率和评估指标。
- 不调用外部 API。

## 推荐任务六：别名映射工具增强

目标：降低 OddsPapi、API-Football 和未来数据源之间的队名差异成本。

范围：

- 别名导入/导出 YAML 或 CSV。
- 低分匹配候选报告。
- 支持按数据源、实体类型、联赛过滤。

验收：

- 使用内存数据库测试。
- 不依赖真实回填数据。

## 推荐顺序

1. 后台 Worker 管理增强。
2. 批次配置执行器。
3. Web 控制台骨架。
4. 盘口结算与训练样本逻辑。
5. 模型训练骨架增强。
6. 别名映射工具增强。

这个顺序能先解决当前操作效率问题，再进入展示和模型能力。

## 公司环境常用验证命令

```powershell
$env:PYTHONPATH='src'
C:\ProgramData\anaconda3\python.exe -m pytest -q
```

如公司电脑没有 Anaconda，可使用可用的 Python 3.11+：

```powershell
$env:PYTHONPATH='src'
python -m pytest -q
```

## 与家里环境的分工

公司环境负责：

- UI、配置、worker 管理、模型骨架、结算逻辑、测试。

家里环境负责：

- 真实 API key。
- 真实 API-Football 同步。
- 真实 OddsPapi 回填。
- 真实 SQLite 覆盖率核对。
- 基于真实样本的大规模训练和回测。
