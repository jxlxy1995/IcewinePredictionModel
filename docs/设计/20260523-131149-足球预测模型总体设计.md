# 足球预测模型总体设计

创建时间：2026-05-23 13:11:49 北京时间

## 一、项目目标

本项目用于构建一个面向主流足球赛事的预测与推荐系统。系统自动拉取比赛基本面数据和赔率走势数据，构建并评估多类模型，输出未来比赛的亚盘和大小球推荐，并给出信心等级与建议出手手数。

第一版目标不是追求全自动投注，而是建立可追踪、可回测、可复盘、可迭代的预测框架。系统可以自动记录推荐结果、自动评估模型表现、自动训练候选模型，但生产模型上线必须经过回测门槛和人工确认。

## 二、赛事范围

第一版覆盖尽量完整的主流足球赛事，参考中国体彩北京单场和主流竞彩的开设口径。

覆盖范围包括：

- 五大联赛及主要杯赛。
- 欧洲次级强联赛，例如德乙、葡超、荷甲、比甲、苏超、土超等。
- 欧冠、欧联、欧协联等欧洲赛事。
- 主要国家队赛事。
- 亚洲、美洲、大洋洲重点联赛，例如日韩澳、美职联、巴甲、阿甲等。

系统维护可配置的联赛白名单。默认排除数据稀薄、盘口不稳定、公开信息不足的地方级别比赛。

## 三、数据源策略

数据源采用“免费优先、付费兜底”的策略。

免费数据优先用于赛程、赛果、基础统计、球队信息、历史结果等内容。付费或半付费数据源用于补充稳定的赔率、盘口、球队统计或历史数据。

付费数据源选择原则：

- 中国大陆访问和支付尽量顺畅。
- 月费不高，适合个人长期维护。
- 提供稳定 API。
- 支持足球赛程、赛果、基础统计、赔率或盘口数据。
- 能够替换，不把业务逻辑绑定到单个供应商。

候选数据源包括 API-Football、The Odds API、football-data.org、Sportmonks 等。最终接入前需要再次确认最新价格、覆盖范围、免费额度、支付方式和中国大陆访问情况。

## 四、技术栈

第一版采用 Python + 数据库 + CLI 的方式实现，同时为后续 Web 可视化预留接口。

技术方向：

- Python 作为核心开发语言。
- 本地 Python 解释器使用 `C:\ProgramData\anaconda3\python.exe`。
- 第一版优先使用 SQLite，后续可平滑迁移到 PostgreSQL。
- CLI 作为第一版主要使用入口。
- FastAPI 作为后续 Web 控制台的数据接口预留方向。
- 后续 Web 控制台用于查看赛程、盘口走势、模型解释、推荐记录、回测表现和赛后复盘。

所有涉及时间的展示、记录、配置和业务判断，均使用北京时间。

## 五、核心模块

### 1. 数据采集模块

负责拉取赛程、赛果、球队、联赛、积分、基本统计、伤停信息和赔率数据。

数据采集模块通过适配器接入不同数据源，每个数据源只负责把外部数据转换为系统内部标准结构。

### 2. 联赛范围模块

维护联赛白名单、联赛优先级、联赛别名和数据源联赛 ID 映射。

第一版不追求覆盖所有低级别赛事，而是优先覆盖主流、有盘口、有稳定数据的比赛。

### 3. 赔率快照模块

系统每次运行都会保存当前亚盘和大小球盘口快照。

同一场比赛可以有多条赔率快照。快照用于分析早盘、中盘、临场盘口变化，并支持之后回放当时的决策环境。

### 4. 特征工程模块

负责从比赛、球队、历史战绩、赔率走势中生成模型输入特征。

第一版重点特征包括：

- 主客队近期战绩。
- 主客场表现。
- 进攻强度和防守强度。
- Elo 或类似强度评分。
- 联赛强度。
- 休息天数和赛程密度。
- 亚盘盘口变化。
- 大小球盘口变化。
- 水位变化。
- 数据完整度标记。

### 5. 模型模块

第一版采用多模型分层融合，不依赖单一模型。

生产模型优先使用可解释、可回测、能直接服务亚盘和大小球的模型。深度学习模型先作为实验层预留，不直接作为第一版正式推荐依据。

### 6. 回测模块

负责按历史时间切分数据，模拟当时可见信息，评估模型表现。

回测必须避免未来数据泄露。所有模型训练和推荐评估都需要记录训练数据截止时间、预测时间和使用的赔率快照时间。

### 7. 推荐模块

负责把模型概率、市场隐含概率、历史同类表现和风险过滤结果转化为亚盘与大小球推荐。

推荐模块输出方向、盘口、水位、信心等级、建议手数、主要理由和风险标签。

### 8. 推荐记录模块

同一场比赛一天内可以多次运行模型并保存多条推荐记录。

开赛前最后一次有效推荐可以标记为最终推荐。赛后结算、盈亏统计和模型评估以最终推荐为主要依据。

## 六、核心数据模型

### 1. 联赛表

保存联赛名称、国家或地区、级别、是否启用、数据源联赛 ID、优先级和别名。

### 2. 球队表

保存球队标准中文名、英文名、数据源球队 ID、所属国家、所属联赛和别名映射。

### 3. 比赛表

保存比赛 ID、联赛、主队、客队、开赛时间、比赛状态、比分、赛果和数据源比赛 ID。

开赛时间使用北京时间。

### 4. 赔率快照表

保存比赛 ID、采集时间、数据源、盘口公司或盘口源、亚盘盘口、主队水位、客队水位、大小球盘口、大球水位、小球水位。

采集时间使用北京时间。

### 5. 特征快照表

保存比赛 ID、特征生成时间、特征版本、赔率快照 ID 和模型输入特征。

特征快照用于复盘当时模型看到的输入数据。

### 6. 模型运行表

保存运行时间、模型版本、训练数据范围、配置参数、训练集截止时间、是否生产版本和运行备注。

### 7. 预测结果表

保存模型原始输出，包括胜平负概率、主客队预期进球、总进球分布、亚盘覆盖概率、大小球打出概率、模型边际优势和校准后置信度。

### 8. 推荐记录表

保存每次运行形成的推荐结果，包括亚盘方向、大小球方向、是否建议出手、信心等级、建议手数、主要理由、风险标签和对应模型运行 ID。

同一场比赛允许有多条推荐记录。

### 9. 最终推荐表

保存或指向开赛前最终确认的一条推荐记录。

最终推荐是赛后结算和模型表现评估的主要依据。

### 10. 赛后结算表

保存实际比分、亚盘结算结果、大小球结算结果、赢半、输半、走水、赢、输、按手数计算的收益和推荐命中情况。

## 七、一次运行的数据流

一次模型运行流程如下：

1. `fetch` 拉取未来比赛、赛果更新和基本面信息。
2. `snapshot` 拉取当前亚盘和大小球赔率，写入赔率快照。
3. `features` 基于比赛信息、历史战绩和赔率走势生成特征快照。
4. `predict` 使用当前生产模型输出概率和优势。
5. `recommend` 将模型输出转为亚盘、大小球推荐、信心等级和手数。
6. `record` 保存本次推荐记录。
7. `finalize` 在开赛前指定窗口内，把最后一次有效推荐标记为最终推荐。
8. `settle` 赛后更新比分和盘口结算。
9. `evaluate` 统计联赛、盘口、信心等级、模型版本的表现。

## 八、模型体系

### 1. 第一版生产模型

第一版生产模型包括：

- 泊松分布模型。
- Dixon-Coles 模型。
- 负二项分布模型。
- 预期进球与丢球模型。
- Elo 或类似球队强度模型。
- 逻辑回归算法。
- 随机森林。
- 梯度提升树模型，例如 LightGBM、XGBoost 或 CatBoost。
- 贝叶斯模型平均或加权 ensemble。
- 概率校准方法，例如 Platt scaling 或 isotonic calibration。

### 2. 比分概率层

比分概率层使用泊松分布模型、Dixon-Coles 模型和负二项分布模型，生成比分矩阵、胜平负概率、总进球概率、亚盘覆盖概率和大小球概率。

Dixon-Coles 模型用于修正低比分相关性，例如 0-0、1-0、0-1、1-1。

负二项分布模型用于处理进球方差比普通泊松更大的场景。

### 3. 球队强度层

球队强度层维护球队进攻强度、防守强度、主客场修正、近期状态衰减权重、联赛强度和 Elo 评分。

该层为比分概率模型和机器学习模型提供基础特征。

### 4. 盘口价值层

盘口价值层使用逻辑回归、随机森林和梯度提升树模型，学习基本面、赔率走势、盘口变化、水位变化与赛果之间的关系。

重点预测：

- 亚盘方向是否有正期望。
- 大小球方向是否有正期望。
- 当前盘口和水位是否存在价值。

### 5. 融合与校准层

融合层不只依赖单一模型，而是综合多个模型输出。

融合方式可以从简单加权开始，后续逐步升级为贝叶斯模型平均。模型权重按回测表现、联赛、盘口类型和样本量进行调整。

所有最终概率都需要进行校准，避免模型长期过度自信。

### 6. 实验模型

循环神经网络、卷积神经网络、生成对抗网络和马尔科夫链模型暂不作为第一版生产推荐依据。

这些模型作为后续实验方向：

- 循环神经网络用于学习赔率时间序列。
- 卷积神经网络用于学习结构化走势矩阵。
- 马尔科夫链模型用于后续临场或滚球状态转移分析。
- 生成对抗网络仅作为研究方向，不进入第一版生产推荐。

## 九、推荐逻辑

推荐不直接由模型倾向决定，而是经过四层过滤：

1. 模型概率。
2. 市场价值。
3. 同类回测。
4. 风险过滤。

只有模型概率明显高于市场隐含概率，历史同类场景表现达标，并且风险标签可接受时，系统才输出出手建议。

### 1. 亚盘推荐

亚盘推荐以当前盘口为准。

系统计算主队方向和客队方向的赢盘、走水、输盘、赢半、输半概率，并结合当前水位计算期望收益。

如果某一方向期望收益明显为正，且通过同类回测与风险过滤，则推荐该方向。否则不出手。

### 2. 大小球推荐

大小球推荐以当前盘口为准。

系统通过比分概率模型生成总进球分布，计算大球和小球打出概率，并结合当前水位计算期望收益。

大小球推荐还需要结合联赛风格、球队节奏、进攻防守强度、盘口变化和水位变化。

## 十、信心等级和手数

信心等级采用 `D / C / B / A / S`，支持 `+/-`。

等级含义：

- `D`：观察，不出手或仅记录倾向。
- `C- / C / C+`：低信心，通常 `0.5 - 0.75 手`。
- `B- / B / B+`：常规推荐，通常 `1 - 1.5 手`。
- `A- / A / A+`：重点推荐，通常 `1.5 - 2 手`。
- `S- / S / S+`：强信心，通常 `2 - 3 手`，极少出现。

手数规则：

- 允许范围为 `0.5 - 3 手`。
- 最小单位为 `0.25 手`。
- 主要出手手数集中在 `1 - 2 手`。
- `3 手` 默认需要极高置信度、优秀同类回测表现和低风险标签。

推荐记录中同时保存信心等级和建议手数。信心等级表达推荐质量，手数表达实际出手强度。二者解耦，便于后续调整仓位配置而不破坏历史评级含义。

## 十一、风险控制

系统默认大量比赛不出手。

触发降级或不出手的典型风险包括：

- 数据缺失。
- 伤停信息不完整。
- 杯赛轮换风险。
- 临场盘口剧烈变化。
- 多数据源赔率冲突。
- 样本量不足。
- 历史同类场景表现不佳。
- 开赛时间过近但赔率更新异常。

系统输出的推荐不是确定性结果，而是基于长期正期望的风险建议。

## 十二、自我学习机制

系统支持自动记录推荐、自动结算、自动统计和自动训练候选模型。

第一版必须保证不会自动把坏模型上线。候选模型进入生产前需要满足：

- 回测样本量达标。
- 收益和命中指标优于当前生产模型。
- 不同时间段表现稳定。
- 不同联赛和盘口类型没有明显灾难性表现。
- 人工确认上线。

## 十三、第一版边界

第一版必须完成：

- 联赛白名单。
- 数据源适配器框架。
- 比赛与赔率快照入库。
- 基础特征工程。
- 泊松、Dixon-Coles、负二项、逻辑回归、梯度提升树等核心模型框架。
- 亚盘和大小球推荐。
- `D/C/B/A/S` 信心等级。
- `0.5 - 3 手`、`0.25 手` 精度的建议手数。
- 推荐记录、最终推荐和赛后结算。
- CLI 使用入口。
- 为后续 Web 控制台预留接口边界。

第一版暂不完成：

- 自动投注。
- 复杂 Web 控制台。
- 深度学习模型生产上线。
- 滚球实时决策。
- GAN 生成式样本正式进入推荐。

## 十四、后续 Web 控制台方向

Web 可视化是后续必要开发内容。

控制台应支持：

- 查看未来赛程。
- 查看赔率历史走势。
- 查看单场模型解释。
- 查看推荐记录和最终推荐。
- 查看赛后结算。
- 查看联赛、盘口、信心等级、模型版本的回测表现。
- 管理联赛白名单和数据源配置。

第一版 CLI 和数据库结构需要为这些能力提前预留，不把逻辑写死在命令行输出中。
