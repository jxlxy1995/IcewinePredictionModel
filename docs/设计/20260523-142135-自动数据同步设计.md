# 自动数据同步设计

创建时间：2026-05-23 14:21:35 北京时间

## 一、目标

第二阶段目标是让系统具备自动拉取数据并入库的能力，减少人工维护成本。

本阶段同时覆盖两类数据：

- 比赛基础数据：联赛、球队、赛程、赛果。
- 赔率快照数据：亚盘和大小球盘口、水位。

第一步采用一个真实 API 做端到端同步。主数据源选择 API-Football，后续预留 The Odds API 作为赔率补充或交叉校验源。

## 二、开发阶段 API 策略

开发阶段先使用 API-Football 免费权限。

官方公开信息显示，API-Football 免费档为每日 100 次请求，并可用于开发验证。该额度足够进行小范围联赛、短时间窗口和低频赔率快照开发，但不适合正式覆盖大量主流联赛和一天多次赔率轮询。

开发阶段限制：

- 默认只启用少量主流联赛。
- 默认同步未来 1 到 3 天比赛。
- 默认每日请求预算为 100。
- CLI 执行前需要估算请求量。
- 超出预算时提示缩小联赛范围、天数或数据类型。
- API 响应需要缓存，避免重复消耗额度。

后续如果进入真实每日使用，再考虑升级到每月 10 到 30 美元左右的付费档。

## 三、数据源架构

系统新增数据源适配器层。

适配器负责把外部 API 字段转换成系统内部标准结构，业务服务不直接依赖 API-Football 的原始响应。

第一版适配器包括：

- `ApiFootballClient`：负责 HTTP 请求、认证、超时、错误解析和请求预算统计。
- `ApiFootballMapper`：负责把 API-Football 响应映射到内部数据结构。
- `ApiFootballProvider`：负责提供同步用的领域方法，例如获取联赛、赛程、赛果和赔率。

后续新增 The Odds API 时，只需要增加新的 provider，不改核心同步流程。

## 四、配置管理

配置采用 `.env + YAML`。

`.env` 只保存敏感信息，不提交到 git：

```env
API_FOOTBALL_KEY=your_api_key
```

`config/sources.yaml` 保存数据源配置：

- 数据源名称。
- API base URL。
- 请求超时。
- 每日请求预算。
- 是否启用缓存。

`config/leagues.yaml` 保存联赛白名单：

- 内部联赛名称。
- 国家或地区。
- API-Football league id。
- 是否启用。
- 优先级。

`config/sync.yaml` 保存同步默认参数：

- 默认未来天数。
- 默认数据源。
- 默认是否同步赔率。
- 默认是否同步赛果。
- 请求预算保护开关。

## 五、联赛白名单

联赛白名单采用“主流联赛 + API-Football 可用 ID”策略。

第一批联赛覆盖：

- 五大联赛：英超、西甲、德甲、意甲、法甲。
- 欧洲重点次级或强联赛：德乙、葡超、荷甲、比甲、苏超、土超。
- 欧洲赛事：欧冠、欧联、欧协联。
- 重点非欧洲联赛：日职、韩职、澳超、美职联、巴甲、阿甲。

北京单场和主流竞彩开设范围作为后续校准方向，不在第二阶段强制完全一致。

## 六、CLI 命令

第二阶段以手动 CLI 触发为主，后续提供 Windows 任务计划程序文档示例。

计划命令：

```powershell
icewine sync leagues
icewine sync upcoming --days 3
icewine sync odds --days 2
icewine sync results --from-date 2026-05-01 --to-date 2026-05-23
icewine sync all --days 3
```

命令行为：

- `sync leagues`：同步白名单联赛基础信息。
- `sync upcoming`：同步未来 N 天赛程、球队和比赛。
- `sync odds`：为未来 N 天已入库比赛拉取赔率快照。
- `sync results`：同步指定日期范围内的赛果。
- `sync all`：按默认顺序执行赛程、赛果和赔率同步。

## 七、入库策略

同步采用 upsert 策略，避免重复创建数据。

唯一识别优先级：

- 联赛：数据源名称 + 外部联赛 ID。
- 球队：数据源名称 + 外部球队 ID。
- 比赛：数据源名称 + 外部比赛 ID。
- 赔率快照：比赛 ID + 数据源 + 盘口源 + 采集时间。

第二阶段需要补充或调整数据库字段，以保存外部数据源 ID 和数据源名称。

## 八、错误处理

典型错误处理：

- 没有 API key：CLI 明确提示配置 `.env`。
- 请求额度不足：终止本次同步并提示缩小范围。
- API 返回限流：记录失败原因，不写入半成品数据。
- 单场赔率缺失：比赛保留，赔率快照跳过并记录日志。
- 联赛不在白名单：默认不入库。
- 网络超时：按配置重试，仍失败则记录错误并继续下一项。

同步日志需要记录：

- 同步开始和结束时间，均使用北京时间。
- 使用的数据源。
- 请求次数。
- 新增和更新的记录数量。
- 跳过和失败的记录数量。
- 失败原因摘要。

## 九、测试策略

第二阶段测试不依赖真实 API key。

测试方式：

- 使用 fixture JSON 模拟 API-Football 响应。
- 测试 mapper 能正确转换联赛、球队、比赛、赔率。
- 测试 provider 在 fake client 下能返回标准结构。
- 测试同步服务能 upsert 数据，不重复创建。
- 测试缺少 API key、超预算、赔率缺失等错误路径。
- CLI 测试使用 fake provider 或 mock 配置，不消耗真实请求。

真实 API 调用只作为手动验证命令，不进入默认自动测试。

## 十、第二阶段边界

本阶段完成：

- `.env + YAML` 配置加载。
- API-Football client、mapper、provider。
- 联赛白名单配置。
- 自动同步未来赛程、赛果和赔率快照。
- CLI 手动同步命令。
- 请求预算保护和同步日志。
- fake response 测试。

本阶段不完成：

- 多数据源交叉校验。
- The Odds API 正式接入。
- 常驻调度器。
- Web 控制台。
- 自动模型训练。
- 正式生产级赔率轮询策略。
