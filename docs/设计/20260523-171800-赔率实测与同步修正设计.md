# 赔率实测与同步修正设计
创建时间：2026-05-23 17:18:00 北京时间

## 一、目标
本阶段验证 API-Football 赔率端点在免费权限下的真实可用性，并修正第一版赔率同步中会影响后续模型分析的几个基础问题：

- `.env` 在 Windows 环境中带 UTF-8 BOM 时仍能读取 `API_FOOTBALL_KEY`。
- 赔率同步只请求北京时间未来窗口内的未开赛比赛，避免浪费 API 请求。
- 从真实返回的多盘口线中，为第一版 `odds_snapshots` 选择一条可分析的亚盘主线和大小球主线。
- API 触发 429 等限制错误时，已经抓取成功的赔率先落库，并返回可读摘要。

## 二、实测结论
使用 2026-05-24 的意甲和西甲赛程做真实请求后确认：

- `odds?fixture=<fixture_id>` 在当前免费权限下可返回数据。
- 多场比赛返回 `results = 1`，每场约 14 个 bookmaker。
- 返回市场中包含 `Asian Handicap` 和 `Goals Over/Under`。
- 单个 bookmaker 会返回多条盘口线，例如亚盘 `Home -1.25`、`Home +0.25`、`Home +1.5`，大小球 `Over 1.5`、`Over 2.5`、`Over 3.0`、`Over 4.5`。
- 连续请求未来 24 小时比赛赔率时可能触发 HTTP 429，因此同步流程需要能保留部分成功结果。

## 三、盘口主线选择
第一版数据库结构每个 bookmaker 的快照只保存一条亚盘线和一条大小球线，因此暂不保存完整盘口曲线。

本阶段策略：

- 同一市场内先按盘口 line 配对。
- 亚盘必须有主队赔率和客队赔率，大小球必须有大球赔率和小球赔率。
- 对每条可配对 line 计算两边赔率差距，优先选择两边赔率更接近的 line。
- 如果差距相同，优先选择平均赔率更接近 2.00 的 line。
- 兼容两类亚盘格式：`Home -0.25 / Away +0.25` 和 `Home -0.25 / Away -0.25`。

这个策略适合第一版模型输入。后续要分析完整赔率走势时，应新增盘口明细表，而不是把多 line 数据压缩进现有 `OddsSnapshot`。

## 四、同步窗口
`sync odds --days N` 使用北京时间当前时刻到 `N` 天后的窗口过滤比赛：

- 只同步 `status = scheduled` 的比赛。
- 只同步 `source_match_id` 不为空的比赛。
- 按开赛时间升序请求。
- 不再扫描全库所有比赛。

## 五、错误处理
API client 将 HTTP 429 等状态码包装为项目内的 `ApiFootballApiError`，并记录本次请求数。

赔率同步改为逐场请求、逐场落库：

- 某场成功后立即保存该场 bookmaker 快照。
- 如果后续 fixture 触发 API 错误，返回已创建数量、已跳过数量、失败 fixture id 和错误摘要。
- CLI 不再直接输出 Python 堆栈。

## 六、后续边界
本阶段不做：

- 完整盘口曲线历史表。
- bookmaker 权重评级。
- 赔率变化趋势特征。
- Web 可视化。

下一阶段可以在已稳定的赔率快照基础上，补充“赔率同步状态查看”和“推荐前特征生成”。
