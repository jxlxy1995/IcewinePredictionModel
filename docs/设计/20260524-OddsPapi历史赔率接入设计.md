# OddsPapi 历史赔率接入设计

## 目标

在不大量消耗付费额度的前提下，验证并接入 OddsPapi 历史赛前赔率，为亚盘和大小球模型训练补充历史盘口样本。

## 约束

- OddsPapi 价格不低，第一阶段必须小样本验证，不能全量回填。
- 免费计划当前显示 `request_limit=250`，历史赔率请求需要严格限额。
- `historical-odds` 一次最多传 3 个 bookmakers。
- OddsPapi 的 `fixtureId` 与 API-Football 的 `fixture_id` 不同，必须先做本地比赛映射。
- 时间统一按北京时间展示，API 请求仍按服务要求使用 UTC ISO 字符串。

## 第一阶段范围

第一阶段只接入：

- 博彩公司：`pinnacle`、`bet365`、`sbobet`
- 市场类型：
  - 全场亚盘：`marketType=spreads`，`period=fulltime`
  - 全场大小球：`marketType=totals`，`period=fulltime`
- 盘口格式：只保留标准盘口 `X`、`X.25`、`X.5`、`X.75`
- 样本规模：最多 20 场本地比赛
- 优先联赛：英超、西甲、意甲、德甲、法甲、德乙、法乙、葡超、中超、J1、K League 1

## 数据流

1. 读取本地 `matches` 中候选比赛。
2. 按联赛、主队、客队、开球时间匹配 OddsPapi fixture。
3. 保存匹配关系到 `odds_source_matches`。
4. 请求 `historical-odds`。
5. 读取或缓存 `markets` 映射。
6. 过滤出全场亚盘和全场大小球。
7. 写入新的历史赔率快照表。
8. 输出样本覆盖报告，不直接进入全量训练。

## 数据库设计

### odds_source_matches

保存本地比赛与外部赔率源 fixture 的映射。

- `id`
- `match_id`
- `source_name`
- `source_fixture_id`
- `matched_at`
- `match_confidence`
- `match_reason`

`source_name` 第一阶段固定为 `oddspapi`。

### historical_odds_snapshots

保存历史盘口快照，和现有 `odds_snapshots` 分开，避免混淆 API-Football 的当前快照。

- `id`
- `match_id`
- `source_name`
- `source_fixture_id`
- `bookmaker`
- `market_type`
- `market_id`
- `market_name`
- `market_line`
- `outcome_side`
- `odds`
- `snapshot_time`
- `period`
- `raw_payload`

`market_type` 使用内部枚举：

- `asian_handicap`
- `total_goals`

`outcome_side` 使用内部枚举：

- 亚盘：`home`、`away`
- 大小球：`over`、`under`

## 成本控制策略

- 所有 OddsPapi 请求都经过 `OddsPapiClient`，统一计数。
- CLI 默认 `--dry-run`，只展示将请求多少场。
- `--max-matches` 默认 20。
- `--request-budget` 默认 50。
- `markets` 映射必须本地缓存，不在每场比赛重复请求。
- fixture 匹配结果缓存到数据库，已匹配比赛不重复请求 fixtures。
- historical odds 已存在时默认跳过，除非显式 `--force`。

## 验收标准

第一阶段完成后，应能回答：

- 20 场样本中有多少场成功匹配 OddsPapi fixture。
- 匹配成功的比赛中，有多少场拿到亚盘。
- 有多少场拿到大小球。
- 每场平均消耗多少请求。
- `pinnacle`、`bet365`、`sbobet` 哪些公司覆盖最好。
- 是否值得购买更高套餐或扩大回填。
