# OddsPapi批量回填控制器设计

## 目标

现有 `oddspapi-fetch` 适合单联赛手动回填，但长批次需要人工盯进度，且无法自动处理超时、限流、连续空数据。新增批量回填控制器，用于把多个联赛按队列自动回填，并在额度充足时通过低并发缩短总耗时。

## 范围

- 第一版只处理 OddsPapi + Pinnacle 历史赔率回填。
- 只运行已配置 OddsPapi tournament 映射的联赛。
- 不修改既有赔率清洗规则，不改盘口主线选择逻辑。
- 不接入 Web 控制台，本功能先作为 CLI 自动化能力。

## 模式

控制器提供三种运行模式：

- `safe`：1 个 worker，5 秒冷却，适合最稳的无人值守。
- `balanced`：2 个 worker，5 秒冷却，默认推荐。
- `fast`：3 个 worker，5 秒冷却，适合额度充足时尝试；遇到异常后仍按联赛任务停住，不无限重试。

## 执行策略

控制器先从 `config/leagues.yaml` 读取启用联赛，按 priority 降序排列，再过滤到 `API_FOOTBALL_TO_ODDSPAPI_TOURNAMENT_IDS` 支持的联赛。也可以通过 CLI 指定 `--league-ids` 限定联赛。

每个联赛任务以小批次循环运行：

1. 每轮最多处理 `chunk_size` 场。
2. 每轮调用既有 OddsPapi 同步逻辑并取得结构化结果。
3. 如果本轮无处理、无写入、无失败，视为无进展，停止该联赛，避免类似未匹配比赛反复消耗时间。
4. 如果本轮处理了比赛但写入 0 条，累计空数据计数；达到阈值后停止该联赛更早日期回填。
5. 如果遇到请求预算耗尽或连续失败超过阈值，停止该联赛。

## 错误处理

- `empty` 和 `unavailable` 仍由既有逻辑作为终态处理。
- `failed` 保留为可重试状态。
- 线程级异常会记录在联赛汇总中，不影响其他联赛。
- 控制器不打印 API key，不读取或输出 `.env` 的敏感值。

## 输出

批量任务输出总览和逐联赛汇总：

- 运行模式、worker 数、总联赛数。
- 每个联赛的状态、轮数、写入快照、处理场次、失败场次、请求数。
- 停止原因，例如候选耗尽、连续空数据、无进展、异常。
