# 胜平负历史赔率训练样本与 Web 曲线设计

日期：2026-05-28

## 背景

隔壁回填线已经把 OddsPapi 胜平负赔率接入 `historical_odds_snapshots`，统一市场类型为 `match_winner`，三项 outcome 为 `home`、`draw`、`away`。模型侧需要让这类数据进入历史赔率训练样本入口，并在 Web 单场赔率走势中展示出来。

同时，Web 赔率曲线之前直接输出数据库中的 `snapshot_time`。OddsPapi 历史赔率快照按 UTC 存储，SQLite 读取后可能丢失时区信息，前端看到的时间容易表现为 UTC。本轮在后端对历史赔率快照时间单独按 UTC 转北京时间输出。

## 样本抽取

`historical_training_sample_service.py` 从两市场扩展为三市场：

- `asian_handicap`：需要 `home` / `away` 成对。
- `total_goals`：需要 `over` / `under` 成对。
- `match_winner`：需要 `home` / `draw` / `away` 三项齐全。

为了兼容三项市场，锚点特征保留原有 `side_a`、`side_b` 字段，并新增可选 `side_c` 字段。对胜平负样本：

- `side_a = home`
- `side_b = draw`
- `side_c = away`

胜平负没有盘口线变化，`market_line` 固定为 `0.00`，`line_movement` 通常为 `0.00`。三项 implied probability 会一起计入 `overround`。

## 结算

`match_winner` 的锚点结算只产生 `win` 或 `loss`：

- 主胜时 `home=win`，`draw/away=loss`。
- 平局时 `draw=win`，`home/away=loss`。
- 客胜时 `away=win`，`home/draw=loss`。

这和亚盘/大小球的走水、半赢半输不同；胜平负本身没有半赢半输。

## Web 曲线

`/api/matches/{match_id}/odds-trends` 新增：

- `match_winner`

每个点包含：

- `snapshot_time`
- `bookmaker`
- `market_line`
- `home_odds`
- `draw_odds`
- `away_odds`

历史赔率快照时间在后端通过专用 formatter 输出为北京时间：

- 如果数据库值带时区，先转 `Asia/Shanghai`。
- 如果数据库值不带时区，按 UTC 解释后再转 `Asia/Shanghai`。

该规则只用于历史赔率快照曲线，不改变比赛开赛时间、推荐记录时间等其他展示时间。

## 边界

- 不修改 OddsPapi 回填器。
- 不修改历史赔率落库结构。
- 不重跑真实回填。
- 不改变旧 `TrainingSample` 赛果样本。
- 不把胜平负接入正式推荐策略；本轮只完成样本入口和曲线展示。
